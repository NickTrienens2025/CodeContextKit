Delivery intelligence brief

DORA metrics for CodeContextKit using only GitHub Issues, PRs, and Releases.

CodeContextKit can start measuring delivery health without adding Jira, incident tooling, or deployment telemetry. Treat GitHub Releases as production deployments, PRs as changes, and Issues as the demand, recovery, and adoption record.

Recommended measurement model

DORA now describes five software delivery performance metrics. For this repo, use the five-metric model but make the first release narrow: delivery timing, release cadence, release failure signals, and adoption.

Timing

Change lead time

For each merged PR included in a release, measure from first commit or PR open time to the GitHub Release `published_at` timestamp.

  • Median and p90 by release
  • Queue time: PR open to merge
  • Release wait: merge to release
Cadence

Deployment frequency

Count published GitHub Releases per week or month. Pre-releases can be tracked separately if they are not the installable tool release.

  • Releases per month
  • Days between releases
  • Version size by PR count
Stability

Change fail rate

Mark a release as failed when a post-release issue is labeled `regression`, `release-blocker`, or `hotfix`, or when a follow-up release explicitly links to a failed release.

  • Failed releases / total releases
  • Failure category labels
  • Hotfix release count
Recovery

Failed deployment recovery time

For release failures, measure from the first failure issue timestamp to the next release that closes or references the recovery issue.

  • Median recovery hours
  • Open failures aging
  • Recovery release link

Adoption layer

DORA says whether delivery is improving; adoption says whether releases matter. For CodeContextKit, pair delivery metrics with release asset downloads, clone/view traffic, install issues, and recurring contributor activity.

  • Release downloads by asset and version.
  • Repository clones and page views over GitHub's rolling traffic window.
  • Issues labeled `install`, `usage`, `feedback`, or `feature-request` after each release.
  • Time from feature issue opened to release containing the closing PR.

Measurement flow

The collector can run on a schedule or after a release. It snapshots GitHub data, normalizes delivery events, computes DORA and adoption metrics, then publishes a static report back into the repository.

GitHub-only implementation

Need
GitHub source
Implementation note
Release/deployment record
REST Releases API: list releases, release assets, `published_at`, tag, target commit.
Normalize every non-draft Release into a `deployments` table. Treat `prerelease` as a separate channel unless you intend it to count.
PR timings
REST Pull Requests API: `created_at`, `merged_at`, commits, labels via the linked issue endpoint.
Map release tags to merged PRs by commit range, release notes references, or closing keywords. Store both PR-open and first-commit lead times.
Issue workflow
REST Issues Timeline API: labels, comments, cross references, close events, and timestamps.
Require lightweight labels: `regression`, `hotfix`, `release-blocker`, `install`, `usage`, `feedback`, `feature-request`.
Adoption
Release assets and repository traffic APIs.
Persist daily snapshots because GitHub traffic endpoints expose a limited recent window. Downloads are cumulative, so calculate deltas.

First dashboard

Start with a generated Markdown or static HTML report in `docs/metrics/`: latest release, releases per month, median lead time, p90 PR age, failed release count, recovery age, download deltas, clone/view deltas, and top adoption issues.

Working agreements

Keep labels strict, link every release to included PRs, require failure issues to reference the release, and snapshot traffic daily. The measurement quality depends more on these habits than on complex analytics.

Bottom line: implement this as a small GitHub data collector plus a static report; use DORA metrics to improve release flow and use adoption signals to decide whether each CodeContextKit release is reaching real users.
