Operating model

Mobile engineering performance from request to production outcome.

Use Jira as the product and work-intake spine, GitLab as the engineering and delivery spine, and a thin metrics layer that joins requests, merge requests, pipelines, mobile releases, production quality, adoption, and AI-assisted work.

Executive intent

The morning view should answer whether important work is moving from request to shipped app capability faster, with acceptable quality, lower production risk, and measurable customer impact. It should not become developer surveillance or a scoreboard of individual activity.

Flow

How fast does important work move?

Measure request-to-production lead time, engineering cycle time, review latency, release wait, and aging work. Break the total time into phases so leaders see the actual constraint.

Quality

Are releases safe?

Pair DORA stability metrics with mobile-specific crash-free sessions, crash-free users, hotfixes, rollback/config mitigations, flaky tests, and pipeline health.

Impact

Are shipped capabilities used?

Track adoption by app version, feature engagement where instrumented, playback or AirPlay success, usage issues, feedback themes, and business-outcome tags.

AI leverage

Is AI improving outcomes?

Measure AI at the work-item and workflow level: cycle-time deltas, review churn, tests added, escaped defects, triage throughput, and developer confidence.

Jira should be the authoritative record for what work is requested, prioritized, shaped, accepted, and validated. Its value depends on clean state transitions and consistent fields, because flow metrics are only as good as the timestamp trail.

GitLab as engineering delivery spine

GitLab should be the authoritative record for branch, merge request, review, pipeline, build, release, and deployment events. DORA metrics and Value Stream Analytics can provide native reference points, while custom extraction fills the mobile-specific gaps.

Required convention

Every branch and MR title must include the Jira issue key. Example branch: mobile-1842-fix-scores-date-scroll-crash. Example MR title: MOBILE-1842 Fix Scores date-scroll crash on iOS 18. Labels should include platform, product area, type, risk, AI involvement, and release train.

Mobile release model

Classic DORA assumes production deployment is clear. Mobile delivery needs a richer event model because merged code may pass through QA, TestFlight, App Store review, phased rollout, user adoption, and meaningful production exposure before the business feels the release.

Dashboard layers

Level 1

Peter morning view

One page for the last 7/30/90 days: request-to-release lead time, active WIP, releases shipped, crash-free sessions/users, change failure rate, top production risks, top flow blockers, AI-assisted work share, code quality trend, and data completeness.

Level 2

Operating review

Product-area views for Scores, GameCenter, VideoCast, AirPlay, Notifications, Auth, Stats, and Platform. Each area gets lead time, open work, aging work, recent releases, crash impact, and AI leverage.

Level 3

Manager and tech-lead drill-down

Issue timelines, MR review timelines, pipeline failures, flaky tests, crash families, release train details, rollout status, and AI-assisted MR outcomes.

Narrative

What Peter should know

Generate a short weekly summary: what improved, what got worse, what is blocked, which risks matter, and what decisions are needed before someone asks.

AI measurement guardrails

Do not measure prompts, AI-generated lines, AI commits, or AI MRs per developer. Those measures are easy to game and do not prove value. Measure AI at the work-item, workflow, and outcome level.

Better metric
Framing
AI-assisted work share
Percent of issues/MRs where AI materially contributed.
AI-assisted cycle-time delta
Compare similar work with and without AI assistance.
AI-assisted review churn
Whether AI-assisted MRs needed more or fewer review iterations.
AI-assisted test coverage
Whether AI helped add tests or reduce escaped defects.
AI-assisted defect rate
Whether AI-assisted changes caused incidents, hotfixes, or severe regressions.
AI workflow reuse
Which repeatable workflows are spreading across teams.
AI confidence rating
Developer self-report: useful, neutral, harmful, or risky.

Rule

AI usage is measured at the work-item level, not the individual developer level. That keeps the dashboard focused on operational leverage, quality, and culture change.

Reference models to copy carefully

GitLab

Copy GitLab’s practical treatment of DORA, Value Stream Analytics, MR analytics, review timing, and pipeline metrics. Do not let MR count or MR rate become a proxy for value.

DORA

Use DORA as the executive vocabulary for delivery performance. Adapt deployment and recovery definitions for mobile because “released” and “used by customers” are different events.

SPACE and EngThrive

Use multidimensional measurement and guardrails. Pair telemetry with survey context so productivity, AI, quality, and developer experience are not collapsed into one fragile number.

Atlassian/Jira

Use cumulative flow and control-chart ideas for bottlenecks, WIP, cycle time, and lead time. Keep workflow hygiene high or the timestamps will mislead.

Executive story

The target story is not “we use AI.” It is: “We have an instrumented mobile delivery system. We can see work move from request to release, understand where time is spent, measure production quality after release, identify crashes by user impact, and use AI to accelerate diagnosis, coding, testing, and reporting without removing human quality gates.”
